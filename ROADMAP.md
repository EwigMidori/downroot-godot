# Downroot Roadmap

本路线图的目标不是罗列全部想法，而是把开发顺序压实，先做出可玩的最小闭环，再逐步扩展系统复杂度。这里有一个重要前提：

- 首版就按 Mod-ready 架构开发。
- 但首版不追求完整的 Mod 产品能力。

也就是说，第一天就要把边界和注册机制设计对，避免未来大改；但不在 `v0.1` 阶段把 Mod 管理、依赖解析、启停切换、兼容策略一次性做完。原则如下：

- 先验证玩法闭环，再补工程化和扩展能力。
- 先做低依赖系统，再做强依赖系统。
- 每个阶段都必须有明确的可试玩验收标准。
- 先把扩展点抽象稳定，再把扩展能力产品化。

## Phase 0: 预生产

目标：搭好能持续开发且可扩展到 Mod 系统的基础骨架，避免后续返工，但不在这一阶段追求玩法完整。

范围：

- 建立 Godot + C# 项目基础结构。
- 明确基础模块边界：
  - Core：事件、配置、输入命令、随机种子、工具类。
  - World：地图、区块、生成、实体放置。
  - Gameplay：玩家、交互、采集、背包、合成。
  - Content：内容定义、注册表、内容包加载。
  - UI：HUD、背包、合成界面。
- 建立内容数据驱动的最小约定：
  - ItemDef
  - RecipeDef
  - PlaceableDef
  - CreatureDef
- 建立统一注册中心：
  - ItemRegistry
  - RecipeRegistry
  - EntityRegistry
  - TerrainRegistry
- 建立内容包概念，即使当前只有 `basegame`：
  - 内容通过 ContentPack 装入，而不是散落硬编码在各模块中。
  - 资源路径、内容 ID、显示名、配方都走统一定义。
- 为未来 Mod 预留扩展接口：
  - `IContentProvider`
  - `IWorldGenPass`
  - `IInteractable`
  - `IDestructible`
- 建立世界生成流水线，而不是把全部生成逻辑写进一个 Godot 节点脚本。
- 建立默认输入绑定，避免硬编码到具体节点脚本。
- 跑通基础资源加载流程，确认现有素材路径可用。

不做：

- 不做完整 Mod 系统产品能力。
- 不做存读档。
- 不做主菜单和设置页。
- 不做高维世界。

验收标准：

- 项目可启动进入一张测试地图。
- 玩家可移动。
- 输入绑定可集中配置。
- 至少一类物品和一类可放置物通过数据定义成功加载。
- `basegame` 以内容包形式注册成功，而不是散落在业务逻辑中。
- 世界生成至少有一层可替换的 pass 接口。

## Phase 1: v0.1 最小闭环

目标：做出第一版真正可玩的生存建造闭环。

架构要求：

- 所有可扩展内容继续走注册表，不允许在 UI 或玩家脚本里写死配方和物品枚举。
- 工作台、掉落物、资源点、敌对生物都必须能通过内容定义和工厂创建。
- 地图生成即使先很简单，也要沿用 Phase 0 的生成流水线。
- `basegame` 视为默认内容包，而不是“特殊逻辑”。

核心循环：

探索地图 -> 收集基础资源 -> 合成工作台/基础物件 -> 建立临时基地 -> 应对夜间威胁

范围：

- 世界
  - 先做有限地图，或固定数量区块。
  - 地面只保留 grass / dirt 两类。
  - 树、石块、蓝莓三类基础资源点。
- 玩家
  - WASD 连续移动。
  - 八格快捷栏 + 十六格背包。
  - 滚轮和数字键切换快捷栏。
- 交互
  - `F` 交互 / 拾取。
  - 左键长按破坏。
  - 右键放置。
  - 最近目标优先，不做“同距离轮转”。
- 生存
  - 昼夜系统先固定时长。
  - 只保留饱腹和生命。
  - 夜间威胁明显，白天威胁降低。
- 资源与掉落
  - 树掉落原木。
  - 地面散落石块可拾取。
  - 蓝莓可直接食用，恢复饱腹。
- 合成
  - 手搓：工作台、火把。
  - 工作台：木箱、木门、木围栏、床。
- 建造
  - 支持放置工作台、木箱、木门、木围栏、床。
  - 大多数放置物可被破坏。
- 敌对生物
  - 只做一种，建议先做蠕虫。
  - 具备简单巡逻 / 追击 / 造成伤害能力。
- UI
  - HUD 显示血量和饱腹条。
  - 快捷栏 UI。
  - 手搓和工作台合成界面。
  - 可交互目标显示 `[F] Interact`。

建议砍掉的内容：

- 不做熔炉。
- 不做矿物团簇。
- 不做工作台升级。
- 不做长矛投掷。
- 不做双击奔跑，改为 `Shift` 冲刺或暂不做。
- 不做首页、加载、Mod 管理、设置页。
- 不做正式的 Mod 扫描、依赖解析、热启停。

验收标准：

- 开新局后，玩家 3 到 5 分钟内可以稳定完成：
  - 收集原木和石块
  - 手搓工作台
  - 放置工作台
  - 制作并放置至少一个基地物件
  - 采集食物并维持生存
  - 在夜晚遭遇并规避或承受敌对生物威胁

## Phase 2: v0.2 资源深化

目标：把“建造生存”推进到“资源加工与工具升级”。

阶段要求：

- 在进入矿物生成和熔炉链路之前，先修正主世界地表生成基线。
- 这次修正不能只是为了让 `grass / dirt` 在当前版本“看起来正常”，而必须考虑后续还要接入：
  - 河流
  - 高于地面层的岩石 / 岩石突起
  - 更复杂的主世界资源散布
  - Phase 3 的区块化与确定性世界扩展
- 因此，地形生成设计必须避免“一次性直接写死最终 tile”的短视做法。
- Phase 2 要先把主世界地表生成重构成可扩展基线；Phase 3 再在这条基线上接河流、地貌和正式世界扩展。

范围：

- 主世界地表生成基线修正
  - 修正 `grass / dirt` 的连续区域分布，不允许随机碎点式混杂。
  - 继续沿用 world gen pass 流水线，不允许退回单脚本硬编码生成。
  - 把“地表基底生成”和“地表资源散布”从职责上分开。
  - 为后续河流和高层岩石预留清晰扩展口。
- 矿物生成系统
  - 普通石块散落生成规则完善。
  - 紫曜石、金沙岩、毒晶石加入主世界。
- 熔炉系统
  - 熔炉放置与交互。
  - 熔炼配方：
    - 紫曜石 -> 紫水晶
    - 金沙岩 -> 黄金 + 沙子
    - 毒晶石 -> 毒水晶 + 铁块
    - 沙子 -> 硅片
- 工具与战斗
  - 斧头
  - 铁刀
  - 近战攻击系统完善
  - 受伤闪红效果
- 敌对生物扩展
  - 加入蟑螂
  - 区分昼夜行为
- 建造扩展
  - 石墙
  - 石地板
  - 木门阻挡敌对生物路径

验收标准：

- 主世界地表已经稳定为可读的连续区域，而不是随机脏点混合。
- Phase 2 的矿物和普通石块散布建立在这套可扩展地表基线上，而不是临时补丁逻辑上。
- 玩家可以从采集基础资源推进到熔炼资源。
- 玩家可以制造至少两种工具并用其提升采集或战斗效率。
- 世界中存在比 v0.1 更明确的中期目标，而不是只靠囤木头生存。

## Phase 3: v0.3 世界扩展

目标：把单地图生存扩展为“探索新区域并获得独有资源”，并在主世界层面正式建立可持续扩展的地貌系统。

范围：

- 主世界从单张固定地图切换到正式区块流送系统。
- 基于世界种子与区块坐标的确定性地图生成。
- 主世界地貌扩展：
  - 河流生成
  - 更稳定的地表区域结构
  - 高于地面层的岩石 / 岩石突起生成
  - 地貌与资源散布规则的分层化
- 区块加载与卸载。
- 传送门生成、激活、过渡表现。
- 高维碎片世界。
- 寒冰晶仅在高维世界生成。

实施顺序：

- Phase 3.1 区块化与生成接口冻结
  - 引入稳定的世界坐标、格子坐标、区块坐标转换规则。
  - 主世界正式运行路径改为“按区块生成并按区块加载”，不保留“单张整图生成”作为正式玩法路径。
  - 生成输入固定为：世界种子、区块坐标、已注册内容。
  - 生成输出固定为：
    - 地表基底
    - 地表覆盖 / 水系
    - 地表区域标签
    - 高于地表层的阻挡型地貌或实体
    - 资源点与特殊位点
- Phase 3.2 主世界区块流送与实体归属
  - 玩家周围按固定加载半径加载区块，离开半径后卸载。
  - 世界实体必须明确归属到区块，并区分：
    - 确定性自然生成内容
    - 运行时变更内容
  - 区块重新加载后，未被玩家改变的自然内容必须稳定复原；已被玩家改变的内容必须在本次运行中保持一致。
- Phase 3.3 主世界地貌 pass 扩展
  - 先生成跨区块连续的地表区域结构。
  - 再生成河流；河流必须写入地表覆盖 / 水系层，而不是回退成“直接写最终 tile”。
  - 再生成高于地表层的岩石 / 岩石突起；这些阻挡型地貌不得与地表层混成一层。
  - 最后生成树、普通石块、矿物、敌对生物和传送门候选位点；这些散布必须依赖前面生成的区域、水系和高层地貌语义。
- Phase 3.4 传送门与高维碎片世界
  - 主世界先生成可发现的传送门位点。
  - 玩家激活传送门后进入独立小世界。
  - 首版高维碎片世界固定为一个由传送门种子决定的 `3 x 3` 区块 pocket world，不做第二套无限世界。
  - 寒冰晶只在该 pocket world 中生成。
- Phase 3.5 收口验证
  - 校验主世界区块边界连续性。
  - 校验河流、岩石突起、资源点和传送门位点在跨区块加载时保持确定性一致。
  - 校验玩家可进入并离开高维碎片世界。
  - 校验寒冰晶只出现在高维碎片世界。

阶段要求：

- Phase 2 必须已经把主世界地表生成改造成可扩展基线。
- 在 Phase 3 中，河流、岩石突起、资源带、传送门生成都必须建立在统一的确定性世界生成口径上，而不是各自单独写一套随机逻辑。
- 主世界生成需要明确区分至少三类概念：
  - 地表基底
  - 地表覆盖 / 水系
  - 高于地表层的阻挡型地貌或实体
- 在进入这一阶段前，先冻结并稳定主世界生成接口。
- 先保证主世界跨区块实体一致性，再接传送门和子世界。
- 不在 Phase 3 内同时引入第二套正式主世界生成路径。
- 不在 Phase 3 内解决存读档持久化；跨运行时持久化放到 Phase 4。

验收标准：

- 主世界已经可以稳定生成河流和高于地面层的岩石地貌。
- 这些地貌在跨区块加载时保持确定性一致，而不是区块边界错位。
- 玩家可以在主世界中发现传送门。
- 进入传送门后可到达一个独立小世界并返回。
- 小世界至少提供一种主世界没有的独占资源。

## Phase 4: v0.4 系统完善

目标：补齐“像一个正式游戏”的基础使用体验。

范围：

- 主菜单
  - 开始游戏
  - 快速开始
  - 详细设置
  - 加载游戏
  - 设置
- 存读档
  - 世界种子
  - 玩家状态
  - 背包
  - 已放置建筑
  - 区块状态
- 调试界面
  - 控制台
  - 常用 debug 开关
- 更多 UI polish
  - tooltip icon
  - 合成高亮反馈
  - 伤害反馈

验收标准：

- 玩家可以创建、保存、加载一个存档。
- 游戏退出重进后世界状态基本一致。
- 调试面板可以支持后续迭代定位问题。

## Phase 5: v0.5 Mod 能力产品化

目标：把前面已经按 Mod-ready 设计的架构，补齐为真正可用的 Mod 产品能力。

范围：

- 多 lib / 多程序集边界整理。
- 内容注册表与加载顺序。
- 基础 Mod 元数据格式。
- 启用 / 禁用 Mod。
- Mod 发现与扫描。
- 基础依赖与顺序处理。
- 内容冲突处理策略。
- 一个官方示例 Mod：传送门 Mod。

前提：

- 核心数据定义必须稳定。
- 世界生成、配方、实体注册接口必须先冻结一版。
- `basegame` 已经证明自己可以完全作为默认内容包运行。

验收标准：

- 不改基础游戏代码，可以通过 Mod 注入至少一类新内容。
- 示例 Mod 可以被启用和禁用。

## 推荐架构草案

下面这套分层是为了兼顾首版开发效率和未来 Mod 扩展，不要求一步做到极致，但建议从一开始就按这个方向建：

### 1. Core

职责：

- 基础配置
- 日志
- 事件总线
- 输入命令抽象
- 随机种子与哈希
- 基础序列化工具

规则：

- 不依赖 Gameplay 和具体内容。

### 2. Content

职责：

- 内容定义类型
- 注册表
- ContentPack 加载
- 内容 ID 与引用解析

规则：

- `basegame` 也是一个 ContentPack。
- UI 和 Gameplay 只依赖已注册内容，不依赖具体资源路径硬编码。

### 3. World

职责：

- 格子与区块数据结构
- 地形层
- 放置物层
- 掉落物层
- 世界生成 pass
- 实体落位与查询

规则：

- 世界生成必须可组合，不要把树、石头、草、矿物生成写死在一个脚本中。

### 4. Gameplay

职责：

- 玩家状态
- 背包
- 交互
- 采集
- 合成
- 战斗
- 生存数值

规则：

- Gameplay 通过接口与注册表拿内容，不直接依赖 `basegame` 的具体类型。

### 5. UI

职责：

- HUD
- 背包
- 合成面板
- 交互提示
- 调试面板

规则：

- UI 只消费 ViewModel 或查询接口，不直接操作世界内部细节。

### 6. Mods

职责：

- Mod 扫描
- 元数据
- 依赖关系
- 启停状态
- 加载顺序

规则：

- 在真正进入 Phase 5 前，这一层可以只有空壳和接口，不必产品化。

## 跨阶段原则

以下问题应尽早定口径，但不要求一开始全部做完：

- 坐标体系：世界坐标、格子坐标、区块坐标之间如何转换。
- 生成口径：哪些内容是纯种子决定，哪些要存档持久化。
- 世界地貌口径：地表基底、水系、地上阻挡地貌、资源点分别属于哪一层，彼此如何叠加。
- 实体口径：地形、可采集物、掉落物、建筑物是否统一成一套实体接口。
- 数据驱动边界：哪些内容可以只改数据，哪些必须写代码。
- 内容 ID 规范：是否采用 `pack_name:item_name` 这样的全局命名。
- 注册时机：引擎启动时、进主菜单时、开局建世界时，各自加载哪些内容。
- 资源覆盖策略：后续 Mod 是否允许 override `basegame` 内容，还是只允许新增。

## 建议执行顺序

如果按实际开发排期，建议按下面顺序推进：

1. Phase 0 中的项目骨架、输入、数据定义。
2. Phase 1 中的玩家移动、交互、背包、采集。
3. Phase 1 中的工作台、放置、HUD、夜晚敌人。
4. 补齐 Phase 1 验收后，再进入 Phase 2。
5. Phase 2 稳定后，再开始 Phase 3 的区块与传送门。
6. Phase 3 开始前，先确认 Phase 2 产出的主世界地表生成基线确实足以承载河流和高层岩石，而不是只够当前矿物散布勉强工作。
7. 最后再补 Phase 4 和 Phase 5。

## 当前版本建议

当前最适合的近期目标是：

- 以 `v0.1.0` 为 Phase 1 目标。
- 只证明“采集 -> 合成 -> 建造 -> 生存压力”闭环成立。
- 所有需要跨系统联动很多模块的内容，都延后到后续版本。
